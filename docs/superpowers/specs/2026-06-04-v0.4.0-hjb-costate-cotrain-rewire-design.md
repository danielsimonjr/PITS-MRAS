# v0.4.0 — HJB/costate co-training rewire (design spec)

**Goal:** Resolve the co-training loop's critic-coupling wiring so the HJB
residual is an *opt-in, actually-applied* critic regularizer, and remove the
vacuous costate-consistency term. Behavior-preserving by default; release v0.4.0.

**Date:** 2026-06-04
**Sub-project:** B of the v0.4.0 set (the others stay queued in `todo.md`).
**Approach:** Approach 1 (reuse `lambda_hjb`, dedicated guarded critic step,
remove costate term + `lambda_costate` field).

---

## Background / problem

In `cotraining_loop` (`src/pits_mras/training/cotrain.py`), `l_total` (the PITNN
objective, stepped by `optimizer_pitnn` which owns **only** PITNN params) used to
accumulate two terms that depend **only on the critic's `W_c`**:

- `loss_cfg.lambda_hjb * l_hjb` where `l_hjb = HJBResidualLoss(critic, e.detach())`
- `loss_cfg.lambda_costate * l_costate` where
  `l_costate = (costate_head(e.detach())[0] - critic.gradient(e.detach()))**2`

Because `optimizer_pitnn.step()` does not own `W_c`, and the IRL block then calls
`critic_optimizer.zero_grad()`, both terms' gradients on `W_c` were **discarded
every step**. (This is the same class of wiring bug fixed for the positivity
regularizer in v0.3.3.)

Two distinct sub-problems were found during brainstorming:

1. **HJB (`l_hjb`)** — a *real, nonzero* critic-training signal (the HJB
   residual; ≈0 at the CARE optimum) that was silently discarded.
2. **Costate (`l_costate`)** — **identically zero** for any `W_c`. `CostateHead`
   has no independent parameters: `CostateHead.forward(e)` returns
   `critic.gradient(e)`, and `grad_V` is the same `critic.gradient(e)`, so
   `l_costate ≡ 0` (verified numerically: `max|λ̂ − ∇V̂| = 0.0`). Identity 2
   (costate = ∇V̂) is enforced **by construction**, so the consistency loss
   measures nothing.

## Decisions (from brainstorming)

- **HJB:** opt-in, default **off**; when enabled, applied as a real gradient to
  the **critic**.
- **Costate:** **remove** the term (it is vacuous; Identity 2 remains enforced by
  construction).
- **Success criterion:** *mechanical correctness* — the HJB gradient is genuinely
  applied to the critic when enabled, the default-off path preserves effective
  behavior, and the critic still converges to `P_opt` with HJB on. No empirical
  "does HJB help" study (the impl plan already notes "no consistent gain").

## Non-goals

- No empirical HJB benefit study / weight tuning.
- No change to `HJBResidualLoss`, `CostateHead`, or `QuadraticCritic` internals.
- No rewire of the *other* v0.4.0 items (dead `LossConfig` fields, H∞ head,
  KKT damped Newton, `ParallelInferenceEngine`, higher-fidelity plants).

## Design

### File structure

| File | Change |
|---|---|
| `src/pits_mras/config.py` | `LossConfig.lambda_hjb` default `0.01 → 0.0`; **remove** `lambda_costate`; update the `lambda_hjb` comment. |
| `src/pits_mras/training/cotrain.py` | Remove HJB + costate from `l_total`; add a dedicated guarded HJB critic step; delete the costate block; drop the `costate_loss` metric; update the module docstring. |
| `tests/test_training.py` | Add HJB-applied-when-on, default-off, and convergence-with-HJB tests; remove/adjust `costate_loss` references. |
| docs / version | `cotrain.py` docstring, dependency graph + `docs/architecture` markers, CHANGELOG, `todo.md`; bump `0.3.3 → 0.4.0`. |

### Co-training loop change

`l_total` becomes a **pure PITNN objective**: `lambda_physics*physics +
(optional) lambda_pcml*pcml + 0.1*cbf`. The HJB and costate blocks are removed
from it.

After `optimizer_pitnn.step()` (alongside the existing positivity critic step),
add a dedicated, opt-in HJB critic step:

```python
# HJB residual regularizer on the critic (opt-in: lambda_hjb > 0). HJB is
# pointwise on e (no IRL-window coupling), so it applies every step when on. A
# real gradient step (not a no-op), so no Adam-step-count bias concern.
l_hjb_val = 0.0
if loss_cfg.lambda_hjb > 0.0:
    critic_optimizer.zero_grad()
    hjb_out = hjb_loss(controller.critic, e.detach())
    (loss_cfg.lambda_hjb * hjb_out["loss"]).backward()
    critic_optimizer.step()
    l_hjb_val = float(hjb_out["loss"].detach())
```

`hjb_loss` stays constructed with `weight=1.0`; `lambda_hjb` is the single knob
applied in the loop. The `hjb_loss` metric records `hjb_out["loss"]` (the
residual MSE), `0.0` when off — preserving the existing `hjb_loss`-metric
semantics (and `test_cotrain_hjb_disabled_path`).

Ordering of critic-optimizer steps per iteration (all on `critic_optimizer`):
1. HJB step — when `lambda_hjb > 0` (real step).
2. Positivity step — when `λ_min(P) < 0` (guarded; v0.3.3).
3. IRL step — when the IRL window is full.

Each is a genuine gradient step toward a distinct objective; none is a spurious
no-op, so Adam's step schedule is not biased.

### Config + backward compatibility

- **`lambda_hjb` 0.01 → 0.0**: preserves *effective* behavior exactly — the term
  was discarded, so the effective critic was already IRL-only. Opt in with
  `lambda_hjb > 0`.
- **`lambda_costate` removed**: a config-API change, but functionally inert (the
  term did nothing). `from_yaml` does `setattr(sub, k, v)` per key, so an old
  YAML carrying `lambda_costate` is **silently ignored** (set-but-unread) — no
  error. Documented as breaking-but-inert in the CHANGELOG.
- **`metrics` dict**: the `costate_loss` key is removed (output-shape change);
  `hjb_loss` is retained.

### Error handling

No new failure modes. `hjb_loss` already handles batched `e`; the guard avoids
computing/stepping when off. The critic stays PD via the (unchanged) positivity
step and IRL warm-start.

## Testing (mechanical correctness)

1. **`test_cotrain_hjb_applied_to_critic_when_enabled`** — isolate HJB as the
   only critic update: `lambda_hjb > 0`, IRL disabled (`irl_window > n_steps`),
   seed a **PD** critic perturbed off `P_opt` (so the positivity step is skipped
   but the HJB residual ≠ 0). Assert `W_c` changes after the loop (gradient
   genuinely applied). Mirrors the v0.3.3 positivity isolation test.
2. **`test_cotrain_hjb_default_off_is_irl_only`** — with default `lambda_hjb=0.0`,
   assert the `hjb_loss` series is all `0.0` (HJB never applied); existing
   IRL-only co-training tests stay green.
3. **`test_cotrain_converges_with_hjb_enabled`** — a `critic_convergence`-style
   run with `lambda_hjb > 0` from a critic seeded off `P_opt`: the relative
   Frobenius error ends below where it started (HJB on does not break IRL
   convergence; near `P_opt` the HJB residual ≈ 0).
4. Remove/adjust any test that asserts the `costate_loss` metric **or sets
   `cfg.losses.lambda_costate`** (e.g. the v0.3.3 positivity isolation test sets
   it; that line becomes dead after the field is removed and should be deleted).
   Identity 2 stays covered by the existing `costate == ∇V̂` test (`test_models` /
   `test_controllers`).

Gates (every commit): `pytest -q`, `flake8 src tests examples
--max-line-length=100 --ignore=E203,W503`, `mypy src/pits_mras
--ignore-missing-imports`.
