# Remove dead LossConfig fields (design spec)

**Goal:** Remove the 6 unconsumed `LossConfig` weight fields (orphaned config
surface). Behavior-preserving; release v0.4.1.

**Date:** 2026-06-04
**Sub-project:** 2 of the remaining v0.4.0 set.

## Decision: remove (not wire)

The 6 fields — `lambda_adjoint` (0.05), `alpha_attn` (0.1), `alpha_smooth`
(0.05), `mu_lyap` (0.01), `beta_param` (1e-4), `lambda_delta_u` (0.01) — have
**zero usages outside `config.py`** (verified). They are dead knobs: setting them
changes nothing.

**Remove**, rather than wire, because:
- Wiring each into its sub-loss (attention entropy, temporal smoothness,
  Lyapunov decay, parameter boundedness, control-rate penalty, adjoint residual)
  is genuine *feature* work that changes training behavior and needs its own
  validation — out of scope for a "remove the dead surface" refinement.
- The sub-loss *classes* mostly already exist (`AttentionRegularizationLoss`,
  `TemporalSmoothnessLoss`, `ParameterBoundednessLoss`, `ControlEffortLoss`,
  `LyapunovConstraintLoss`) and expose their **own** weights, so the capability
  is available to anyone who wants it — the orphaned config fields add nothing.
  (`lambda_adjoint` has no implemented loss at all.)
- Keeping non-functional config knobs misleads users (anti-pattern). Removal is
  behavior-preserving (they're unused) and the honest reduction of surface.

If a specific sub-loss is later wanted in the active `TotalLoss`/training loops,
that is a focused feature with its own design — not a resurrected dead field.

## Blast radius

- `src/pits_mras/config.py` — delete the 6 fields from `LossConfig`.
- `tests/test_config.py::test_loss_config_defaults` — delete the 6 assertions.
- Docs mentioning the fields in LossConfig field lists: `docs/architecture/API.md`
  (`lambda_adjoint=0.05` in the defaults line), `docs/architecture/ARCHITECTURE.md`
  (LossConfig field list includes `lambda_adjoint`), `docs/architecture/COMPONENTS.md`
  (LossConfig field list). Update those lists.
- No `_COMPONENTS` entry references any of the 6 (they were never wired), so
  `TotalLoss` is untouched.

## Backward compatibility

Config-API change (field removal). `from_yaml` does `setattr` per key, so an old
YAML carrying any of these keys loads without error (set-but-unread). No
migration needed.
